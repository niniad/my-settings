# Amazon Creative Workflow

Amazon商品ページのサブ画像・A+コンテンツを、AIを活用して設計・生成する。

## ワークフロー全体像

```
Phase 1: 全体訴求マップ設計（何を・どの順で・どのメディアで伝えるか）
    ↓
Phase 2: 個別画像のレイアウト・モジュール設計（ワイヤーフレーム）
    ↓
Phase 3: 素材準備（写真レタッチ・AI素材生成）
    ↓
Phase 4: デザイン生成（HTML/CSS → PNG変換）
    ↓
Phase 5: レビュー・修正
```

各フェーズは独立して実行可能。「訴求マップだけ」「ワイヤーフレームから生成して」など部分的リクエストにも対応する。

---

## Phase 1: 全体訴求マップ設計

商品の訴求ポイントを整理し、サブ画像（最大8枚＋動画1-2本）とA+コンテンツの全体構成を決める。

**入力**: 商品情報（商品名、特徴、ターゲット、競合との差別化）
**出力**: 訴求マップテーブル

**詳細 → `references/phase1-appeal-map.md`**

### 訴求マップのテーブル形式

| 枠番号 | 配置場所 | メディア種別 | 訴求タイプ | 訴求内容（要約） | 優先度 |
|--------|----------|------------|----------|----------------|--------|
| 1 | サブ画像1枚目 | 静止画 | メイン商品 | 白背景商品写真 | 必須 |
| 2 | サブ画像2枚目 | 静止画 | 権威付け | 保育士推薦 | 高 |
| ... | ... | ... | ... | ... | ... |
| V1 | サブ画像枠 | 動画 | 使用シーン | 食事中の使用風景 | 高 |
| A1 | A+モジュール1 | A+画像 | 詳細解説 | 素材の特徴 | 中 |

### メディア割り当て基準

- **動画優先**: 動作を伴う機能（ポケットを開く等）、使用シーンのリアル感、Before/After
- **サブ画像優先**: 感情訴求、権威付け、第一印象キーメッセージ（スクロール中に目を止める役割）
- **A+画像優先**: 詳細スペック、FAQ補足、ブランドストーリー、じっくり読む系情報
- **A+比較表優先**: 競合比較、バリエーション一覧、スペック表

### サブ画像の鉄板構成パターン

```
1枚目: メイン画像（白背景・商品のみ）
2枚目: 権威付け or 強力なキャッチ
3枚目: お悩み提起（ターゲットの痛みに共感）
4-6枚目: 機能訴求（お悩みへの解決策）
7枚目: 商品詳細（スペック・寸法図）
動画: 使用シーン or 機能デモ
```

あくまでベース。商品特性に応じて調整する。1画像あたりの訴求は最大3ポイントに抑える。

## Phase 2: レイアウト・モジュール設計

各画像のワイヤーフレームを設計し、A+はモジュール選択も行う。

**入力**: Phase 1の訴求マップ、ベンチマーク画像（任意）
**出力**: 各画像のワイヤーフレーム＋テキスト原稿

**詳細 → `references/phase2-layout-design.md`**

### 画像サイズの決定

画像サイズはカテゴリにより最適解が異なる。デフォルトは以下:

| タイプ | サイズ | 用途 |
|--------|--------|------|
| 正方形 | 2000×2000px | サブ画像（多くのカテゴリで推奨） |
| 縦長 | 1600×2000px | サブ画像（情報量多いカテゴリ） |
| A+標準 | 970×600px | A+ 標準画像＋テキスト |
| A+比較表 | 各150×300px | A+ 比較表の商品画像 |

出品前にBSR上位10商品のサブ画像サイズを確認し、カテゴリの主流に合わせる。

### A+モジュール選択基準

**詳細 → `references/amazon-aplus-modules.md`**

| 訴求内容 | 推奨モジュール |
|----------|--------------|
| 複数機能の並列説明 | スタンダード4つの画像とテキスト |
| 競合比較・スペック比較 | スタンダード比較表 |
| 大きな写真で魅せる | スタンダード画像とテキストのオーバーレイ |
| 詳細なテキスト説明 | スタンダードテキスト付き画像(左/右) |
| ブランドストーリー | ブランドストーリー |

### ベンチマーク画像の検索

NocoDB APIでタグベース検索を行う。検索クエリの例:

```
レイアウト = "grid_comparison" AND 訴求タイプ = "problem_agitation"
→ お悩み×グリッド比較の構図ベンチマーク候補を取得
```

取得した画像を参考に、レイアウトの構成要素（グリッド数、テキスト位置、写真サイズ比率）を決定する。

## Phase 3: 素材準備

デザインに必要な写真・イラスト素材を準備する。

**入力**: Phase 2のワイヤーフレーム、商品写真原本
**出力**: レタッチ済み写真、生成素材

**詳細 → `references/phase3-material-prep.md`**

### 処理カテゴリ

1. **商品写真レタッチ**: 背景除去（rembg）、色調補正、影付け、リサイズ・トリミング
2. **ライフスタイル素材生成**: AI画像生成で使用シーン写真を作成（洗濯機、泣いている赤ちゃん等）
3. **アイコン・バッジ生成**: ✕/◎アイコン、POINTバッジ、装飾要素
4. **手書き風テキスト画像**: HTMLで再現困難な手書き風テキストは画像として生成

---

## Phase 4: デザイン生成

HTML/CSSでデザインを組み立て、Puppeteerで高解像度PNGとして出力する。

**入力**: Phase 2のワイヤーフレーム、Phase 3の素材
**出力**: 完成画像（PNG）

**詳細 → `references/phase4-design-generation.md`**
**レンダリング → `scripts/render_html.py`**

### HTML/CSSベースの利点

- 日本語テキストが100%正確（画像生成AIのような文字化け・脱字がない）
- フォントサイズ・ウェイトの正確な制御（情報ヒエラルキーの表現）
- テキスト修正が即座に可能（全体を再生成する必要がない）
- テンプレート化で同シリーズ商品に横展開可能
- Puppeteerで任意の解像度のPNG出力が可能

### HTMLで再現が難しい要素への対処

| 要素 | 対処方法 |
|------|----------|
| 手書き風フォント | Google Fontsの手書き系フォント or 画像生成 |
| 有機的な装飾（キラキラ等） | SVG or 装飾素材をPNG生成して`<img>`配置 |
| 複雑なグラデーション | CSS gradient or 背景画像として生成 |
| 斜めのテキスト配置 | CSS transform: rotate() |

## Phase 5: レビュー・修正

生成されたデザインの品質チェックと修正。

### チェック項目

1. **テキスト視認性**: スマホサイズ（375px幅相当）で文字が読めるか
2. **情報ヒエラルキー**: 見出し→本文→キャプションのサイズ差が明確か
3. **カラーバランス**: ブランドカラーの統一性
4. **Amazon準拠**: ガイドライン違反がないか
5. **訴求密度**: 1画像あたり3ポイント以下か
6. **A+サイズ要件**: モジュールの画像サイズ要件を満たしているか

### 修正は部分的に行える

```
テキスト修正 → HTMLのテキスト編集 → 再レンダリング（数秒）
レイアウト修正 → CSS調整 → 再レンダリング
写真差し替え → <img>のsrc変更 → 再レンダリング
```

## ベンチマーク画像管理

NoCoDBで管理。タグ体系の詳細 → `references/benchmark-tagging.md`

### 最小限のタグ（まずこの2軸で始める）

**レイアウト構造**: `hero_text_side` / `grid_comparison` / `numbered_list` / `spec_diagram` / `endorsement` / `variation_gallery` / `before_after` / `feature_callout`

**訴求タイプ**: `problem_agitation` / `feature_benefit` / `competitive_advantage` / `authority` / `spec_detail` / `usage_scene`

### ベンチマーク登録フロー

1. 画像をNoCoDBのAttachmentフィールドにアップロード
2. レイアウト構造タグを付与
3. 訴求タイプタグを付与
4. 任意でメモ（カラー、フォント特徴、参考元ASINなど）を記入
